一个成功的Git分支模型--A successful Git branching model

发表于 2017-01-03 | 分类于 Dev

本文是蹩脚的中文翻译版，这是原文地址

本文中我讲展示一个Git开发模型，过去一年中我的所有项目(工作和个人)都在使用，已经被证明是非常成功的。很长时间以前我就想写这些，不过直到现在才有时间。我不会讨论项目的详情，仅仅只是介绍分支策略和发布管理。

上图主要是关于Git如何用来做代码版本管理工具。(顺便提一下，如果你对Git感兴趣，我们公司GitPrime提供了非常棒的软件工程性能方面的实时分析系统)

为什么选择Git

关于Git和其他集中式源码管理工具相比的优缺点的深入分析，看这里和这里。这方面的争论之战不休，但是作为一个码农，相比其他工具，我更喜欢用Git。Git确实改变了开发者关于分支和合并的思考方式。当年我们还在用经典的CSV／Subversion时，分支/合并总是让人担惊受怕(“合并冲突时小心点，否则有你好看的”)，所以很长时间内你只会做一次

但是对于Git，这些操作就变的非常简单加方便，并且会被认为是你日常工作中的核心部分。比如，在一些CVS/Subversion的书中，分支和合并总是会在最后的几章才出现(针对高级用户)。但是在任何一本介绍Git的书中，基本上第三章以前就会覆盖到了。

Git这种简单和可重复的特性使分支和合并不再让人担心，相比其他功能，分支和合并应该是版本控制工具最重要的部分。

关于版本管理用具的部分就不啰嗦了，我们来关注下开发模型。我要介绍的这个模型，无外乎就是项目开发团队中所有成员都要遵守的开发方式。

分布式而非集中式

我们先假设使用一个运转良好的中心”真实的”版本库。注意，这个版本库只被认为是一个中心版本库(不过由于Git是分布式的，所以原则上来讲是没有中心版本库这种东西的)，我们将这个库命名为orgin，相信Git的用户对这个名字都很熟悉。

每个开发人员从origin中推送和拉取代码，但是除了和集中式的推送拉取关系，每个开发人员可能都需要从其他子团队的节点中获取改动。比如有时两个以上的开发者完成一次大的新特性开发时，为了不过早的将代码发布到origin, 就需要采用这种方式。如上图，有Alice和Bob,Alice和David，Clair和David几个子团队。

技术实现上，就是Alice新建一个分支，命名为Bob，将分支指向Bob的版本库，反之亦然。

主分支

核心部分，开发模型主要式如图所示的模型，中心库由以下两个无限延伸的分支构成：

  • master
  • develop

Git用户应该对origin上的master分支很熟悉，与master分支平行的另一分支式develop分支.

origin/master上的HEAD节点的代码一直都是可以发布状态。
origin/develop分支上的HEAD节点的代码则是在上次发布后的修改，准备下一次发布。有些人会将它称作是”集成分支”，某些公司夜间自动构建的代码就是源自这个分支。

当develop分支上的代码达到可以发布的稳定状态后，所有的改变需要以某种方式合并到master分支，打上版本号。具体的细节将会在以后讨论。

所以，每次合并改动到master分支时，实际上就定义了一个新的发布版本。所以每次我们向master上commit时都要格外小心，技术上，我们可以用Git Hook脚本来自动的构建和发布软件到生产环境。

辅助分支

除了master和develop分支，我们的开发模型使用了一系列的辅助分支用来支持团队成员间并行开发，方便功能追踪，做发布准备，快速修复生产环境问题等。和主分支不同的是，这些分支的生命周期是有限的，因为它们最终将会被移除。

我们使用的不同类型的分支有：

  • 功能分支
  • 发布分支
  • 快速修复分支

这些分支都有特定的目的，而且它们起始于哪个分支，将合并到哪个节点，都要受限于严格的规则。我们将马上演练。

技术层面讲，这些并不是”特殊”的分支，而只是普通的的Git分支，它们的类型取决于我们怎么使用。

功能分支

可能是起始于develop的某个分支
必须要合并回develop分支
命名规则: 不能使用master, develop, release-*, or hotfix-*

功能分支(又是也被称作Topic分支)通常用来开发下一个版本或是未来某个版本的新功能。开始开发新功能时，无法确定开发完成后将纳入此功能的发布节点。所以功能分支将会存在，直到功能开发完成后合并到develop分支(以便最终纳入发布版)或功能被丢弃(以防效果让人失望)。

功能分支通常只在开发人员的库中，不会在origin库中

发布分支

可能起始于develop分支
必须要合并回develop和master
分支的命名: release-*

发布分支是为新产品发布做准备的。发布分支允许在产品上线前的最后一刻做小的bug修复和准备发布的元数据(版本号，构建日期等等)。通过在发布分支做这些操作，可以使develop分支在下次面临大的特性发布时保持清晰。

创建发布分支的时机，是develop分支(几乎)已经达到可以发布的状态了。至少所有的将发布的新功能此刻已经合并到develop分支。在创建发布分支之后，其他需要在未来版本中增加的功能才能提交到develop分支。

在发布分支创建时才为即将进行的发布分配版本号，不可能更早了。因为直到那一刻，develop分支所反映的变化，都是针对“下一次发布”的，但是还不能确定“下一次发布”的版本号将会是0.3还是1.0。直到发布分支创建之后，才能按照版本号规则确定版本号。

这个发布分支会一直存在，直到新的版本能稳定运行。期间可以在发布分支(而不是develop分支)上面修复一些新版本的bug。应该严格禁止在发布分支上面作大的功能性开发，只能在develop分支上进行，然后等待下次发布。

快速修复分支

可能起始于master
必须要合并回develop和master
命名规则: hotfix-*

快速修复分支和发布分支非常像，都是为最新的发行版准备的，不同的是快速修复分支是未经计划的。它们被用来紧急修复不稳定的发行版，比如一些发行版的眼中bug，快速修复分支一般是由master分支上发行版对应的tag创建。

其本质是团队成员可以继续工作(在develop分支上)，而另一个人为发行版准备一个快速的修复。

创建快速修复分支

总结

尽管这个模型并提出没有什么让人意外的新东西，但是本文开始的图片中的模型，已经在我们的项目中被证明是非常有用的。他构造了一个优雅的思维模型，能让团队成员很容易的理解分支和发布流程。
